Architecture and Scope

Ampra Internal Slack Knowledge Agent

Prepared by Richie for Julien and Dustin. A first-pass architecture for an agent that answers employee questions in Slack from Ampra's SOPs.

The objective

A Slack agent that answers team questions instantly using Ampra's own SOPs in Google Shared Drive, so people self-serve instead of interrupting each other or digging through Docs. It should answer only from approved SOPs, cite the source, and hand off cleanly when it does not know.

Questions to lock before building

The fastest way to waste a build is to guess these. A few answers from you two and the scope tightens immediately:

  • Which Shared Drive or folders are the source of truth, and roughly how many docs?
  • How often do the SOPs change? (this sets the re-index cadence)
  • Trigger preference: mention the bot, a /ask slash command, or a dedicated #ask-ampra channel?
  • Strict mode (answers only from SOPs, always cites) or also allowed to give general help?
  • When it has no answer, where should it route the person? (a human, an #ops channel)
  • Confirmed that Ampra owns the Google Cloud project, Slack app, and hosting?

Recommended architecture

Two pipelines, both on reliable, established tooling. AI is used only at the answer step, with guardrails. This is a retrieval augmented generation (RAG) setup, deterministic where it can be.

1. Ingestion (scheduled + on demand)

1
n8n lists SOP docs in the Shared Drive via the Google Drive API
2
Pulls each doc's text via the Google Docs API, chunks it (about 500 to 800 tokens, with overlap)
3
Embeds the chunks and upserts them into a vector store (Supabase pgvector recommended: low cost, Ampra-owned, simple. Pinecone is the alternative)
4
Stores metadata with each chunk: doc title, link, last modified. Re-indexes nightly plus a manual trigger

2. Answering (live)

1
Slack Events API fires on a mention or /ask command into an n8n webhook
2
The question is embedded and run against the vector store, top 4 to 6 matching chunks returned
3
Claude answers using only those chunks, with a system prompt that forces it to cite the source SOP and link, and to say it does not have it documented rather than guess
4
The answer posts back in the Slack thread with source links. No match means it routes to the human or channel you choose

Why this stack (the reliability question)

Established orchestration (n8n) plus a managed vector store plus Claude is dependable and maintainable in a real business. Experimental agent frameworks tend to break in production. The pipeline stays deterministic, and AI only runs at the generation step where it is wrapped in guardrails. This is the difference between a demo and something the team can actually rely on every day.

Reliability and guardrails

  • Strict prompt: answer only from retrieved SOPs, always cite the source, never invent
  • No-match fallback that escalates to a person or an ops channel instead of guessing
  • Every question and answer logged. Bonus: that log shows the most-asked questions, which surfaces exactly which SOPs are missing or unclear and should be written next

Ownership, security, maintainability

  • Ampra owns the Google Cloud project, Slack app, vector store, and n8n instance and all credentials. No lock-in.
  • Least-privilege access: a read-only service account scoped to the SOP Shared Drive only
  • Clear naming on every workflow and node, with notes inside explaining what each does
  • A one-page runbook: how to add an SOP, how to rotate keys, what not to touch
  • Secrets in the credential store, never hardcoded

Phasing

  • Phase 1 (MVP): one Shared Drive, mention trigger, top-k retrieval with citations, manual re-index. Prove it works on a seed set of SOPs.
  • Phase 2: slash command and dedicated channel, nightly auto re-index, no-match escalation, thumbs up/down feedback on answers.
  • Phase 3: a most-asked-questions report that flags SOP gaps. This turns the agent into an insight tool, not just a lookup.

How I would build it

Claude Code with skills and memory files to scaffold the n8n workflows, the retrieval logic, and the guardrail prompt quickly, then test against a seed set of SOPs before it touches the team. Idea to working draft fast, then hardened into something reliable.
